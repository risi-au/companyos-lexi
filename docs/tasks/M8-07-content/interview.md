## Interview guide

Work through these areas in whatever order the conversation flows; skip what the
pack already answers. Depth follows the interviewee — brief answers get follow-ups,
detailed answers get structured.

1. **Outcome** — what does success look like for this scope, concretely? By when?
   How will it be measured (metrics, deliverables, revenue)?
2. **The work** — what actually gets done here, by whom, how often? One-off build
   or ongoing operation?
3. **History** — if the pack included related history (e.g. a sales trail), confirm
   what was promised/quoted and what carries over as commitments.
4. **Systems** — which external tools does this scope touch (CRM, email, hosting,
   ads, accounting, analytics)? What already exists vs needs setting up?
5. **Access** — which credentials will agents need (names + what-for only, never
   values)? Who currently holds them? Any access do's/don'ts worth writing into a
   connection doc?
6. **Provisioning** — does this scope need task tracking (Plane)? A code workbench
   (repo)? An agent token from day one?
7. **Starting state** — what should exist the moment the scope is provisioned:
   first documents, first two weeks of tasks, facts the wiki should know.
8. **Risks & unknowns** — what could sink this? What couldn't the interviewee
   answer?

## Packet instructions

Return your markdown brief, then end with the single fenced JSON packet exactly as
specified in the operating guide. Facts labelled as facts, assumptions as
assumptions, unknowns in `open_questions`. No secret values anywhere.
