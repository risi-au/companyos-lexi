# M7-02 pre-implementation analysis

1. V1 should accept GitHub `ping`, `push`, and `pull_request`, but only turn PR actions `opened`, `reopened`, `synchronize`, and `closed` with `merged: true` into workbench events. Other PR actions such as review requests, labels, assignments, comments, and ready-for-review would add noise without improving the safety net.
2. Workbench resolution should match `repoFullName` first, then normalize paths and choose the deepest `workbenches.path` prefix per changed file. Root-level workbench rows use an empty path and catch root files only when no deeper prefix matches. Nested scopes win over ancestors. Renamed files should consider previous and current filenames so both old and new folders can be resolved. A single push can therefore produce multiple scope groups, one per deepest matched scope, without merging sibling paths.
3. Duplicate suppression needs two layers: GitHub delivery id prevents retries from emitting another event or record, while PR merge stub creation checks recent records and session completion events in the resolved scope for the same PR URL/number or commit SHA/range. That keeps normal `log_change` / `complete_session` as the primary path and makes GitHub ingestion a fallback.
4. CompanyOS should store only durable work-record metadata: repo, branch, PR number/url/title, author login, commit SHAs/range, changed path samples, resolved scope path, delivery id, and a changelog stub when no agent wrap-up exists. GitHub remains the source for files, diffs, full commits, reviews, comments, and secrets.
5. The leak risk is resolving by repo alone or by a non-deepest/root prefix and then recording sibling/client paths in the wrong scope. Payloads must be grouped by matched scope and path samples must be limited to paths belonging to that scope group. Unmatched paths should stay in an ignored/null-scope event or response, not be attached to another tenant/client scope.
