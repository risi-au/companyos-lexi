# CompanyOS — Design Document (v1)

*The founding design. Ratified 2026-07-02 through a full requirements + architecture dialogue between Rishi and Claude (Fable 5). Supersedes everything in `/legacy`.*

---

## 1. What CompanyOS is

**A self-hosted, AI-native system of record and visibility layer for running businesses.** Work happens elsewhere — in terminal agents, vendor chat tools, ad platforms, workbench folders. CompanyOS is the one place that *knows everything*: what exists, what changed, what's pending, what it cost, and why.

Built as a **SaaS whose first tenant is the holding company**. Internal use first; external tenants later; the architecture assumes multi-tenant-by-instances from day one.

### The core bet

> **Chats are disposable. Tools are disposable. The record is the asset.**

Agents and vendor tools are stateless workers. All durable state — tasks, changelogs, decisions, reports, metrics, knowledge — lives in CompanyOS as structured, scoped, queryable data. Any agent, any vendor, today or in five years, gets context from the OS and reports outcomes back to the OS. Switching tools costs muscle memory, never data.

## 2. Requirements contract (ratified)

1. **Scope tree** — arbitrary depth. Top level = client or project (`airbuddy`, `indya`, `llmtxt`); below that, anything (`indya/marketing/meta-ads/retargeting`). Every node is a scope; the root is the company-wide view.
2. **Modules per node, composable** — dashboard, canvas, KB docs, tasks, metrics, capabilities registry, changelog. Attach/remove per scope at any time. Modules are plugins over a stable kernel: nothing is rigid.
3. **Agent-authored dashboards** — dashboards are stored specs rendered by the shell. Agents create/edit them via MCP ("add a spend-by-country widget"). Every level has one, including the root home view. Widget vocabulary is extensible; escape hatch for custom widgets.
4. **Tasks** — for humans (Rishi + future team), not an agent work queue. Agents create/close tasks via MCP as they do work (terminal agent finishes website fix → task auto-closes). Replaces Trello. Delegated to Plane (see §3).
5. **Record store** — changelogs, decisions, analysis reports per scope, markdown bodies. This is the durable memory. Vendor chat tools pull context at session start (`get_context`) and write structured exhaust at session end.
6. **MCP/API front door** — every agent surface (Claude Code, Grok, Hermes, Cursor, Flowise, n8n) connects through one MCP server: `get_context`, `log_change`, `complete_task`, `save_dashboard`, …
7. **Metrics store** — generic time-series (scope, metric, date, value, dimensions). Filled by scheduled n8n pulls (Meta, Google, GA4, WooCommerce/Shopify) or manual/agent writes. Dashboards and alert agents query it. Optional per scope.
8. **Workbenches** — local folders + git where actual work happens. One repo per top-level scope, folders mirror the tree; dedicated repos as exceptions (code-heavy nodes). `AGENTS.md` per workbench is generated by the OS and wires agents to the MCP with the right scope key. GitHub = truth for code/files; OS = truth for the record; the scope key joins them. Structure sections in MDs are OS-generated between markers; drift is detected by diffing registry vs repo.
9. **Capabilities registry** — agents/automations built in any engine (n8n, Flowise, Hermes, custom), registered per scope with their own scoped token. OS shows status + run history, deep-links to the engine for edits. Alerting/monitoring = just another registered capability. Per-step AI-vs-deterministic choice belongs to the builder, not the platform.
10. **Skills** — one central git repo, agentskills.io `SKILL.md` format (Hermes-verified compatible; Claude Code native). Frontmatter declares scope pattern + domain tags. Resolution = global + matching domains + scope branch, specific overrides general. Delivered by workbench sync (local agents) and `list_skills` MCP (remote).
11. **Permissions** — humans and agents are the same thing: **principals** with per-scope **grants**. Built day one; admin UI when the team arrives; client-facing portal possible later without rearchitecting.
12. **Onboarding** — 80% deterministic provisioning (`provision_scope`: registry, repo skeleton, module instances, tokens — idempotent code), 20% judgment (intake conversation, brief writing, dashboard drafting — frontier model). Incremental: "add SEO to indya" is the same call.
13. **Self-hosted, zero lock-in** — VPS, Docker Compose, all data in own Postgres + git. Every adopted component swappable behind the OS contract.
14. **Resident agent** — always-on chat inside the OS, per-scope and global. The guaranteed-connected surface; the universal inbox (paste any external conversation/doc → structured OS actions); quick scoped Q&A. Model + reasoning selectable per conversation via the gateway. Its continuity comes from the record store, never private memory — deleting it loses nothing.
15. **AI-maintainable codebase** — see CONSTITUTION.md. Kernel + vertical modules, event bus, per-module AGENTS.md, mechanical boundary enforcement. An agent improving one module never reads the whole codebase.
16. **SaaS structure** — two admin planes. **Control plane** (platform super-admin): tenant registry, provision/suspend/remove tenant instances, per-tenant resource + token usage, versions, platform roles, billing hooks later. **Tenant admin** (module inside each tenant): user management, SSO (OIDC via Better Auth), agent token management, observability — API usage + token spend by scope/capability/model/principal, full activity log (the events table), admin-gated MCP tools (`query_events`, `query_usage`) so optimization agents can analyze the OS's own usage.
17. **Portability** — API-first: web UI, MCP agents, and future mobile (Expo/React Native) are three clients of one typed service layer. Web ships as responsive PWA day one. Nothing reachable only through the UI.

**Terminology:** a SaaS customer = **tenant** (tenant #1 = the holding company). Inside a tenant, airbuddy/indya are **scopes**. Never conflate.

## 3. Adopt-vs-build map (ratified)

| Piece | Verdict | Choice | Notes |
|---|---|---|---|
| Tasks/PM | Adopt | **Plane CE** (self-hosted) | Native MCP server, open API, webhooks, RBAC, Docker Compose. Open-core: verify CE feature boundaries during build. OS maps tree→Plane (project/labels), proxies task MCP tools, ingests webhooks into events |
| Automation | Adopt | **n8n** (self-hosted) | Schedules, data pulls, webhooks, AI nodes. ⚠️ Sustainable Use license — review before first external tenant |
| Visual agent builder | Adopt | **Flowise** | Registered as capability engine |
| Canvas | Adopt (embed) | **Excalidraw** React component | Scenes = JSON in our Postgres; agent-editable |
| KB editor | Adopt (embed) | **BlockNote** | Notion-style out of box; core liberally licensed. **Markdown is canonical storage**; editor constrained to markdown-representable blocks; docs sync to workbenches as `.md`. Plane's wiki rejected: HTML storage, page-edit API immature (issue #7319), and the KB is crown-jewel data the OS must own |
| Model gateway | Adopt | **LiteLLM** (self-hosted) | One OpenAI-compatible endpoint. Direct provider keys (Anthropic, DeepSeek, Google, …) + OpenRouter as upstream for exotics. **Virtual keys per capability = per-scope cost tracking + hard budget caps.** Role aliases (`cheap`/`analysis`/`reasoning`) — workflows never name vendors. Subscriptions (Claude Max etc.) stay in their own vendor tools, which connect via MCP — two lanes, no conflict |
| Database | Adopt | **Postgres** (+pgvector later) | One DB server; separate logical DBs per app (os, plane, n8n, litellm) |
| Charts | Adopt (embed) | React chart lib inside widget vocabulary | Not Grafana/Metabase — the dashboard layer must be agent-authored |
| Shell (tree, modules, permissions, UI) | **Build** | | The product |
| Record + metrics stores | **Build** | | Crown jewels |
| Dashboard spec + renderer | **Build** | | Nothing off-the-shelf is agent-authored |
| MCP server | **Build** | Official TS SDK | Thin layer over the same service layer as the UI |
| Resident agent | **Build (thin)** | | Chat UI + system prompt + MCP tools + gateway. No framework |
| Provisioning | **Build** | | Deterministic, idempotent |
| Control plane | **Build (thin v1)** | | Tenant registry + usage over instance-per-tenant model |

**Dropped from the legacy plan:** gbrain (record store absorbs it; re-attachable later as a search capability or via pgvector module — KB is markdown, which gbrain ingests, so the door stays open), Hindsight (learnings → records + skills), Affine (canvas → Excalidraw embed, docs → BlockNote; application-vs-library principle), Discord as primary surface (optional notification target for alert capabilities).

## 4. Stack (ratified)

- **TypeScript end-to-end**, monorepo (pnpm workspaces + turborepo)
- `apps/os` — the tenant product: **Next.js** (App Router), **shadcn/ui + Tailwind** (design tokens; theming/white-label via token sets; modules compose shared primitives only)
- `apps/control-plane` — thin Next.js app
- `packages/db` — **Drizzle ORM** schemas + migrations
- `packages/api` — the typed service layer (all business logic; UI and MCP both consume it)
- `packages/mcp` — MCP server on the official TypeScript SDK (stdio + HTTP transports)
- `packages/ui` — shared primitives + design tokens
- **Better Auth** — sessions, orgs/roles, OIDC/SSO-ready, mobile token flows
- **Deployment:** Docker Compose (Caddy TLS + os + control-plane + Postgres + Plane + n8n + Flowise + LiteLLM), 12-factor env config, automated migrations + seed. One-command fresh-instance boot = the DR story and the SaaS install
- **Mobile later:** Expo/React Native consuming `packages/api` client + tokens via NativeWind. PWA in the meantime

## 5. Data model (kernel contract)

Kernel (stable, everything depends on it):
- `scopes` — id, parent_id, path (`airbuddy/marketing/meta-ads`), name, type, status, settings jsonb
- `principals` — kind (`human`|`agent`), name, contact
- `grants` — principal × scope × role (owner/admin/editor/viewer/agent); a grant covers the subtree
- `tokens` — hashed key → principal, expiry, last_used; every MCP call authenticates → principal → grants
- `module_instances` — scope × module type × config jsonb × order
- `events` — append-only: type, scope, principal, payload, ts. **Every write emits one.** Audit trail + module message bus + activity feed + usage telemetry source

Modules (each owns its tables; no cross-module table access):
- `records` — scope, kind (changelog|decision|report|note), title, body_md, data jsonb, author, links
- `documents` + `document_revisions` — scope, title, body_md (markdown canonical), synced to workbenches
- `canvases` — scope, name, scene jsonb
- `metrics` — scope, metric, date, value, dims jsonb; unique(scope, metric, date, dims_hash)
- `dashboards` + `dashboard_revisions` — scope, spec jsonb (versioned, revertable)
- `capabilities` + `capability_runs` — scope, name, engine, engine_ref, token_id; runs: status, timestamps, summary
- `workbenches` — scope → repo + path; drives MD generation + folder↔OS join
- `task_links` — scope ↔ Plane project mapping; OS↔Plane issue refs (Plane owns task data)
- `skills_index` — cache of skills repo: name, scope_pattern, domains, path (git = source of truth)

Properties: markdown bodies wherever content lives; jsonb where flexibility matters; events on every write.

## 6. MCP contract v1

Every call: token → principal → grants → scope check → execute → emit event. Additive versioning only (new tools / deprecation windows; never silent changes).

- **Context:** `get_context(scope)` (identity + brief + recent changes + open tasks + skills index, token-budgeted), `search(scope, query)`, `get_tree(scope)`
- **Records:** `log_change`, `log_decision`, `save_report`, `list_records(scope, kind, since)`
- **Tasks (Plane proxy):** `create_task`, `update_task`, `complete_task`, `list_tasks`
- **Docs:** `save_doc(scope, title, markdown)`, `get_doc`, `list_docs`
- **Dashboards:** `save_dashboard(scope, spec)`, `get_dashboard`, `list_widget_types`
- **Metrics:** `write_metrics(batch)`, `query_metrics(scope, names, range, group_by)`
- **Canvas:** `save_canvas`, `get_canvas`
- **Skills:** `list_skills(scope)`, `get_skill(name)`
- **Admin-gated:** `provision_scope(spec)`, `register_capability`, `report_run`, `query_events`, `query_usage`

## 7. Build order

- **M1 — Spine:** monorepo scaffold, kernel schema, service layer + auth, MCP server (context/records/tasks), Plane deployed + adapter, LiteLLM deployed. *Proof: terminal agent in a workbench closes a task + logs a change via MCP; Trello retired.*
- **M2 — Visibility:** metrics store, first n8n pulls (one real client), dashboard spec + renderer + starter widgets, agent-authored dashboards. *Proof: the AirBuddy overview dashboard.*
- **M3 — Knowledge & front desk:** KB (BlockNote, md-canonical, workbench sync), canvas, resident agent with model picker. *Proof: onboard indya from a pasted transcript.*
- **M4 — Automation depth:** provisioning automation (repo skeletons, MD generation), capabilities registry + run reporting, skills integration, alert pattern.
- **M5 — SaaS hardening:** tenant admin (users, SSO, observability UI), control plane v1, one-command Docker bundle, backups + DR drill.

## 8. Risks / checkpoints

- n8n Sustainable Use license — legal read before first external tenant (engine is swappable behind the capabilities registry if needed)
- Plane CE vs paid feature boundary — verify during M1
- BlockNote markdown round-trip fidelity — constrain block set in M3; test lossiness early
- Scope-tree ↔ Plane-hierarchy mapping (tree is deeper than Plane's model) — adapter design in M1
- Token/cost runaway — LiteLLM budgets per virtual key from day one
